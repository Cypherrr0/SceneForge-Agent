================================================================================
检索系统优化策略 - 提升Hit@1和Recall@3
================================================================================

【当前性能】
- 重排Hit@1: 45.08% (119/264)
- 重排Recall@3: 64.02% (169/264)
- 融合Recall@10: 91.29% (理论上限)

【问题诊断】
1. 111个case正确答案不在Top3（真正的检索失败）
2. 30个case正确答案在第2位（平均差距0.90分）
3. 13个case正确答案在第3位（平均差距0.95分）
4. 23个case融合检索Top10就找不到正确答案（8.71%）

【优化策略】按预期效果排序

================================================================================
策略1：增加重排候选数量 (预期提升Recall@3: +27个百分点)
================================================================================
目标：提升Recall@3从64.02%到91.29%

实施方法：
在 test_complex_query_v3.py 第1057行：
  fusion_top_k=20  改为  fusion_top_k=50

原理：
- 融合检索Recall@10=91.29%，说明正确答案大多在Top10-50之间
- 增加候选可以让重排模型有更多选择机会
- 虽然会增加10-20ms的重排时间，但Recall@3可大幅提升

预期效果：
- Recall@3: 64.02% -> ~85-90%
- Hit@1: 45.08% -> ~50-55%（间接提升）

================================================================================
策略2：调整融合检索权重 (预期提升Hit@1: +3-5个百分点)
================================================================================
目标：给向量检索更高权重，提升融合质量

实施方法：
在 test_complex_query_v3.py 第1036行，fusion调用处添加参数：
  fusion_results = reciprocal_rank_fusion(
      bm25_optimized_results, 
      vector_results,
      bm25_weight=0.3,    # 从0.4降到0.3
      vector_weight=0.7   # 从0.6升到0.7
  )

原理：
- 向量检索Hit@1=32.20% > BM25的20.45%
- 向量检索更能理解语义，应该有更高权重
- 测试数据显示向量检索MRR=0.433 > BM25的0.306

预期效果：
- 融合Recall@10: 91.29% -> ~93-95%
- Hit@1: 45.08% -> ~48-50%

================================================================================
策略3：改进查询表达 (预期提升Hit@1: +5-8个百分点)
================================================================================
目标：让查询更明确，减少歧义

典型问题和修改建议：

1. 问题：curve.shade_smooth vs object.shade_smooth (3个错误)
   原查询："ops set smooth shading on object"
   改进为："ops object set smooth shading"
   
2. 问题：fisheye_lens vs lens (1个错误)
   原查询："types set camera lens focal length"
   改进为："types camera lens focal length property"
   
3. 问题：view_camera vs camera_add (3个错误)
   原查询："ops insert camera view"
   改进为："ops add camera object to scene"

4. 问题：transform_apply vs resize (多个错误)
   原查询："ops resize object scale"
   改进为："ops transform resize scale"

实施方法：
修改 queries.json，在查询中：
- 明确添加类别前缀（object/mesh/curve）
- 使用更具体的术语（add而非insert，property而非value）
- 包含更多上下文关键词

预期效果：
- Hit@1: 45.08% -> ~52-55%
- Recall@3: 64.02% -> ~70-75%

================================================================================
策略4：优化重排模型输入格式 (预期提升Hit@1: +2-3个百分点)
================================================================================
目标：改进查询-文档对的构造方式

当前问题：
- 文档被截断到400字符，可能丢失关键信息
- 查询没有预处理，直接送入重排模型

实施方法：
在 test_complex_query_v3.py 的 rerank_candidates 函数（第505行）：

当前代码：
  if len(doc) > 400:
      doc = doc[:400] + "..."

改进为：
  # 提取API签名和描述的关键部分
  doc_lines = doc.split('\n')
  api_signature = doc_lines[0] if doc_lines else doc  # API名称行
  doc_desc = '\n'.join(doc_lines[1:])
  if len(doc_desc) > 350:
      doc_desc = doc_desc[:350] + "..."
  doc = api_signature + '\n' + doc_desc

预期效果：
- 保留完整API签名，提高匹配准确度
- Hit@1: 45.08% -> ~47-48%

================================================================================
策略5：调整RRF融合参数k (预期提升Hit@1: +1-2个百分点)
================================================================================
目标：优化倒数排名融合的平滑程度

当前设置：k=60（第1036行的reciprocal_rank_fusion调用）

实施方法：
测试不同的k值：
- k=40：更激进，强调前几位结果
- k=80：更保守，考虑更多长尾结果

推荐尝试k=40，因为：
- BM25和向量检索的Top5差异较大
- 较小的k会让高排名的结果更有区分度

在调用处添加：
  fusion_results = reciprocal_rank_fusion(
      bm25_optimized_results, 
      vector_results,
      k=40  # 从默认60改为40
  )

预期效果：
- Hit@1: 45.08% -> ~46-47%
